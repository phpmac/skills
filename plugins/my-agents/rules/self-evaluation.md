# 任务自评估规范

## 触发条件

任何完成的任务(代码审查/方案设计/bug修复/审计/报告)都必须执行自评估.

## 评估方式

完成后写出评判过程, 回答以下问题:

### 我检查了哪些位置?

列出本次任务实际检查/修改的所有文件/位置/逻辑路径.

### 我可能遗漏了什么?

诚实列出本次任务可能未覆盖的部分:
- 哪些边界条件没有测试?
- 哪些关联位置没有检查?
- 哪些假设没有验证?

### 我的结论有哪些不确定的地方?

标注置信度:
- **高置信**: 有代码/数据/搜索结果直接支撑
- **中置信**: 基于经验推断, 未直接验证
- **低置信**: 猜测或类比, 需要人工确认

### 下一步建议

如果时间允许, 最应该补充检查什么?

## 输出格式

在产出末尾追加:

```
## 自评估

**已检查**: <列出位置/路径>

**可能遗漏**: <列出未覆盖部分, 无则写"无明显遗漏">

**不确定项**: <列出低/中置信结论, 标注原因>

**建议补充**: <最值得补充的一项>
```

## 与 distill 的关系

自评估中的"可能遗漏"和"不确定项"是蒸馏的高优先输入: 分析为什么遗漏, 提炼为规则防止再犯.
