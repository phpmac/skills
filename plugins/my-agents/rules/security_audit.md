# 安全审计完整性规范

- 安全审计中发现的漏洞必须经过实际验证, 严禁使用虚构/推测的 PoC 数据
- 链上漏洞必须附带真实的链上调用命令(如 cast call)和实际返回数据, 禁止编造链上返回值
- 安全审计必须采用两种思维: (1) 管道式审计 - 沿单条业务流程从头到尾研究每个环节的逻辑, 理解流程设计后推导出可能的问题面(如提现自然会有签名绕过/重入/金额校验/状态不同步等问题); (2) 交叉式审计 - 分析不同业务流程之间的交叉影响(如充值的状态变更是否影响提现判断). 不能只检查单函数/单接口, 必须理解完整流程再审计
- 如实报告未发现漏洞的情况, 禁止为了填充报告而夸大漏洞严重程度或编造发现
- 漏洞严重程度评级必须基于实际可利用性, 禁止仅凭理论推断就标记为 Critical/High
