# 安全审计完整性规范

- 安全审计中发现的漏洞必须经过实际验证, 严禁使用虚构/推测的 PoC 数据
- 链上漏洞必须附带真实的链上调用命令(如 cast call)和实际返回数据, 禁止编造链上返回值
- 安全审计必须采用两种思维: (1) 管道式审计 - 沿单条业务流程从头到尾研究每个环节的逻辑, 理解流程设计后推导出可能的问题面(如提现自然会有签名绕过/重入/金额校验/状态不同步等问题); (2) 交叉式审计 - 分析不同业务流程之间的交叉影响(如充值的状态变更是否影响提现判断). 不能只检查单函数/单接口, 必须理解完整流程再审计
- 如实报告未发现漏洞的情况, 禁止为了填充报告而夸大漏洞严重程度或编造发现
- 漏洞严重程度评级必须基于实际可利用性, 禁止仅凭理论推断就标记为 Critical/High
- 漏洞分析/类似漏洞搜索/攻击案例查询优先使用 x-search (推特安全研究员的 AI 处理结果), GitHub 仓库目录爬取使用 firecrawl
- [近期流行]审计代币合约时必须检查0xdead销毁地址相关的三个风险点: (1) 购买到销毁地址的绕过路径 - swap(to=0xdead)/transfer(to=0xdead)是否绕过买入限制/手续费/黑名单等检查; (2) 底池价格操纵风险 - 大量代币流入0xdead导致AMM底池代币余额骤降价格异常飙高, 闪电贷可放大此效果; (3) 0xdead在白名单中时, 分析其具体影响 - 0xdead在白名单中本身可能是合理的(燃烧需要), 不能直接标高风险, 关键是分析: 哪些检查被跳过, 跳过后的攻击路径是否成立
