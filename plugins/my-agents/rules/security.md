# 安全规范

## 开发阶段

- 项目有提现功能时, 测试环境必须禁止提现
- 上线前必须进行真实线上环境验证和安全验证
- 用户资产余额数据库(migration)禁止为负数, 使用 unsigned
- 禁止 VITE_ 开头的环境变量存储敏感信息(会泄漏到前端, 特别是私钥)

## 审计方法

- 如实报告未发现漏洞, 禁止夸大或编造
- 漏洞必须经过实际验证, 严禁虚构/推测 PoC 数据
- 链上漏洞必须附带真实 cast call 命令和实际返回数据
- 严重程度评级基于实际可利用性, 禁止理论推断标 Critical/High
- 漏洞分析/攻击案例查询优先使用 x-search, GitHub 仓库爬取使用 firecrawl
- 两种审计思维: (1) 管道式 - 沿单条业务流程从头到尾研究每个环节; (2) 交叉式 - 分析不同流程间的影响
- 代币合约必须检查 0xdead 销毁地址三个风险点: (1) swap/transfer 到 0xdead 是否绕过限制; (2) 大量代币流入 0xdead 导致 AMM 价格操纵; (3) 0xdead 在白名单中的具体影响分析

## 审计交付标准

| 类型 | 标记 | 可交付 |
|------|------|--------|
| 静态分析 (Slither/Mythril) | `## 未验证发现` | 否 |
| 符号执行/模拟交易 | `## 待验证` | 否 |
| 链上 cast call 验证通过 | `## 已验证 [severity]` + 实际返回值 | 是 |
| forge test PoC 通过 | `## 确认可利用` + 测试输出 | 是 |
| 无法验证 | `## 无法验证, 疑似风险` | 用户决定 |

硬规则: 禁止编造 PoC 数字 / 禁止未验证标 severity / 必须区分未验证 vs 确认可利用
