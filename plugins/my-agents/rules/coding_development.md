# 编码/开发规范

- 辅助使用 context7 解决代码API/语法/方法问题
- 修改业务逻辑前,先全局搜索同类实现,确保统一修改
- 同一业务逻辑出现2次以上,立即提取公共函数,不询问确认
- 调用链关系:追溯上游调用方和下游依赖,评估改动影响范围
- 修改文件之前一定要先看看类似文件的代码实现,确保风格修改
- 方法内部逻辑:逐行阅读理解实现细节,识别边界条件和状态变更
- 业务上下文:理解代码要解决的实际问题,确保修改符合业务预期
- 测试覆盖:检查现有测试用例,确保修改后测试仍能覆盖核心逻辑
- 一个问题解决了很久才解决的要记录到项目的 `docs/常见问题.md` 文件中
- 有很多需要修改的时候是不能直接修改的,得提供修改前后对比预览,同意后才能修改
- 编写之前不懂的或者过时的就使用 context7/firecrawl 获取最新信息,不要随意乱编写
