# vibe/代码/编码/开发规范

- 禁止使用cat代替文件写入操作
- 执行包管理器命令前必须先确认项目使用的是 npm/yarn/bun/pnpm, 禁止假设
- Shell 命令中注意引号转义, 特别是远程执行和 fish shell 环境下
- 我默认使用的是 fish 而不是 bash/sh/zsh
- 代码commit提交必须要使用对应框架格式化代码
- 编写代码之前需要设计可靠方案,避免低效率和重复
- 关键的工具/命令/环境缺失必须停止并报告交给人工处理
- Skill/Agent 触发时必须先检查 metadata 或文档中声明的工具依赖, 逐个验证是否已安装, 未安装必须报告并停止
- 同一业务逻辑出现2次以上,立即提取公共函数,不询问确认
- 限制代码长度一般不超过500-1000行,使用拆分和组件化来解决
- 实在无法解决的就使用流程图呈现,结合关键代码片段由人工提供思路来解决
- 一个问题解决了很久才解决的要记录到项目的 `docs/常见问题.md` 文件中
- 业务上下文:理解代码要解决的实际问题,确保修改符合业务预期,只看相关部分
- 状态变更/恢复/重置类操作必须做边界检查, 包括防重复执行/前置条件校验/数据一致性校验, 确保操作幂等性
- 状态/类型/分类等固定取值字段必须使用语言原生枚举机制定义, 禁止字符串/数字硬编码
  - PHP: 使用 enum
  - TypeScript: 使用 const 对象 + as const 或 union type
  - Python: 使用 StrEnum 或 Literal
  - Go: 使用自定义类型 + const iota
  - 核心原则: 固定取值的集合必须收敛到单一声明处, 通过引用使用, 不得分散在各处硬编码
- 特别注意在 plan 模式下,没有提供方案和人工确认的情况下是禁止直接修改代码的
- 不要频繁的执行命令,要想好,确定方案再整理命令
- 通常需要使用npm的都会使用bun代替
