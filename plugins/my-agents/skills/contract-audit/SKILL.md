# 合约审计

你正在帮助用户对智能合约进行安全审计. 遵循以下三阶段流程.

## 核心原则

- **工具预检**: 执行前必须检测 agent 需要的工具是否已安装且配置正确, 否则禁止执行后续操作, 安装方法在 agent 里面有说明,没有的话根据当前系统环境结合mcp工具查阅安装方法
- **边界全覆盖**: 审计目标不能只看单个合约, 必须递归分析所有交互合约 (如 Pair, Router, Factory, Proxy 等)
- **深度代码阅读**: 发现问题时必须仔细阅读该方法的完整内部实现及所有关联调用链, 禁止仅凭函数签名或表面逻辑下结论
- **先规划后执行**: 必须先输出审计方案并获得用户确认, 才能启动 agent 执行
- **使用 TaskCreate**: 全程跟踪审计进度

---

## 阶段 1: Agent 选择

**目标**: 让用户选择本次审计使用的 agent

**行动**:

1. 读取 `${CLAUDE_PLUGIN_ROOT}/agents/` 目录, 列出所有安全相关 agent:

| Agent | 触发关键词 | 职责 |
|-------|-----------|------|
| **contract-scanner** | Slither, Mythril, 工具扫描 | 静态分析/符号执行/fuzzing |
| **smart-contract-vuln** | DeFi 漏洞, 攻击模式 | 漏洞知识库 + 代码逐项审计 |
| **vuln-discovery-orchestrator** | 综合审计, 全面审计 | 协调多个 agent 分配任务并汇总 |
| **vuln-taxonomy-researcher** | 漏洞分类, 检查清单 | 生成针对性检查清单 |
| **poc-verifier** | PoC, 漏洞验证 | 将疑似漏洞转化为可复现 PoC |
| **framework-auditor** | Web 框架, CVE | 前后端框架安全 (非合约) |

2. 使用 AskUserQuestion 让用户选择要使用的 agent (multiSelect), 默认推荐 `vuln-discovery-orchestrator` (综合审计)
3. 如果用户选择单个 agent, 直接使用; 选择 orchestrator 则由它自动委派

---

## 阶段 2: 审计方案

**目标**: 生成详细审计方案, 包含边界分析, 获得用户确认

**行动**:

### 2.1 项目扫描

1. 确定审计目标路径 (用户指定或当前目录)
2. 扫描所有 Solidity 文件, 识别:
   - 主合约列表 (名称, 类型, 文件路径)
   - 继承关系 (is, inherits)
   - 外部依赖 (import, interface)
   - 已部署合约地址 (如果用户提供)
3. 递归追踪合约交互链, 例如:
   - Token -> Pair (sync/skim 边界)
   - Router -> Pair (流动性操作边界)
   - Factory -> Pair (创建逻辑边界)
   - Proxy -> Implementation (代理模式边界)
   - MasterChef -> RewardToken (奖励分发边界)
   - LendingPool -> InterestRate (利率模型边界)

### 2.2 生成审计方案

输出格式:

```
## 审计范围

### 核心合约
- [合约名] (文件路径) - [类型: Token/Pair/Router/...]

### 关联合约 (边界审计)
- [合约名] - [与核心合约的关系] - [风险点]

### 外部依赖
- [库名/接口] - [用途]

## 审计维度

### 1. 常规漏洞检查
- 重入攻击 (Reentrancy)
- 整数溢出/下溢
- 访问控制缺失
- 闪电贷攻击面
- 价格操控
- 前端交易 (Front-running)

### 2. 边界审计 (根据关联合约)
- Pair.sync() / Pair.skim() 资产提取风险
- Router 代币转入授权边界
- Factory createPair 唯一性保证
- Proxy delegatecall 存储冲突
- [根据实际合约补充的边界项]

### 3. 业务逻辑审计 (流程审计)

**核心原则: 必须先理解业务功能的完整流程, 再审计每个环节是否存在漏洞. 不能逐个函数孤立审查, 要串联整个链路分析.**

#### 审计方法论: 两种思维

**思维1: 管道式审计 (纵向 - 单条流程)**
- 沿着单条业务流程从头到尾完整研究, 把流程当成一条管道
- 先梳理完整链路: 如充值流程 = 用户发起 -> 后端校验 -> 合约接收 -> 记录余额 -> 事件通知
- **理解流程设计后推导问题面**: 了解流程逻辑后, 自然能推导出该流程可能存在的问题类型
  - 提现流程: 签名校验绕过(谁能拿到签名)/重入(回调再入)/金额校验(是否允许0或超额)/状态不同步(后端与链上)
  - 充值流程: 回调重入/金额不一致(实际到账与记录)/重复充值(同一笔交易重复记账)
  - 质押流程: 凭证伪造/收益计算精度/解质押锁定绕过
  - 交易/兑换流程: 闪电贷操控价格/滑点保护绕过/三明治攻击
  - 清算流程: 闪电贷操纵抵押率/清算顺序依赖/自清算套利
- 再逐环节研究每个节点的逻辑是否存在推导出的那些问题
- 最后将所有环节串联起来整体验证: 管道是否有缝隙, 数据从入口到出口是否一致

**思维2: 交叉式审计 (横向 - 流程间交叉)**
- 分析不同业务流程之间的交叉点和相互影响
- 如充值流程的状态变更是否会影响提现流程的判断逻辑
- 两条管道的交叉点是否存在状态冲突, 权限泄露, 数据不一致
- 重点关注: 共享状态/共享权限/共享资金池的业务流程之间的影响

典型流程示例:
- 充值: 用户发起 -> 后端校验 -> 合约接收 -> 记录余额 -> 事件通知
- 提现: 用户申请 -> 后端签名 -> 合约验证签名 -> 转账执行
- 质押: 存入资产 -> 获取凭证 -> 累积收益 -> 解质押提回

审计要点:
- 流程中每个环节的权限传递是否合理(如签名是否可被非预期方获取)
- 流程步骤是否可以跳过或重排(如未验证直接执行后续步骤)
- 跨合约/跨系统的状态一致性(如后端状态与链上状态是否同步)
- 流程中的信任假设是否成立(如后端是否信任了不应信任的输入)
- 不同业务流程的交叉点是否存在状态冲突或权限泄露

## 执行计划
- Agent: [用户选择的 agent]
- 步骤: [具体执行顺序]
```

3. 使用 AskUserQuestion 让用户确认方案, 或提出修改意见
4. 用户确认后进入阶段 3

---

## 阶段 3: 执行审计

**目标**: 调用选定的 agent 执行审计方案

**行动**:

1. 根据用户选择的 agent, 构造 agent 提示:
   - 包含完整的审计方案内容
   - 指明审计目标路径和合约范围
   - 包含边界审计的具体检查项
   - **强调深度阅读**: agent 必须完整阅读每个可疑函数的内部实现代码, 包括其调用的所有内部方法和外部合约方法, 追踪完整的数据流和状态变更, 禁止仅看函数签名或注释就下结论
   - **强调流程审计**: agent 必须先梳理每个业务功能的完整流程链路(管道式), 再分析不同流程间的交叉影响(交叉式), 禁止跳过流程理解直接逐函数审计
   - 要求 agent 返回结构化的审计报告

2. 对于 `vuln-discovery-orchestrator`:
   - 直接传入完整审计方案, 让它自动分配子任务
   - 等待完成后汇总所有子 agent 结果

3. 对于单个 agent:
   - 传入审计方案中的对应维度
   - 如果需要多个维度, 可并行启动多个 agent 实例

4. 汇总所有 agent 结果, 输出最终审计报告:

```
## 审计报告

### 摘要
- 审计范围: [合约数量, 文件数]
- 发现问题: [高危/中危/低危/信息] 数量统计

### 发现详情

#### [高危] [问题标题]
- 合约: [合约名::函数名]
- 描述: [问题说明]
- 影响: [实际影响评估]
- 建议: [修复方案]

### 边界审计结果
- [边界检查项及结果]

### 总体评估
- [安全评级和总结]
```

5. 标记所有 TaskCreate 完成

---

## 注意事项

- 审计报告中的影响评级必须基于实际可攻击性, 禁止理论推断
- 如果 agent 报告工具未安装, 立即停止并报告给用户
- 边界审计中如果发现新的关联合约, 需要追加到审计范围并告知用户
- 所有回复和注释使用中文, 标点符号使用英文标点
