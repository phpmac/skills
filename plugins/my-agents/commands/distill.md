## 规范

- 只蒸馏方法/流程/规范/技巧/思路/经验
- 如果对话中没有可蒸馏内容, 告知用户暂无
- 禁止直接写入, 必须先展示完整内容经用户确认

## 蒸馏优先级

1. **纠正/踩坑** -> Rule 或 CLAUDE.md (最高优先, 防止再犯)
2. **可复用工作流** -> Skill
3. **全局行为规范** -> CLAUDE.md 追加

## 蒸馏流程

### 第一步: 会话分析

回顾对话, 按三维度提取:

| 维度 | 关注点 |
|------|--------|
| 成功 | 什么方法有效/发现了什么模式/解决了什么难题 |
| 失败 | 什么方法无效/走了什么弯路/根因是什么 |
| 发现 | 新的洞察/意外的边界情况/配置相关发现 |

### 第二步: 深度提炼

- 一次只问一个问题, 像聊天不像问卷
- 优先从踩坑/失败开始挖掘, 挖不到再退到成功案例
- 追问"为什么这样决定"而不是只问"做了什么"

### 第三步: 分类与封装

按优先级选择载体:

| 纠正类 | 提炼为规则: 做什么/不做什么/为什么 |
|--------|-----------------------------------|
| 工作流类 | 提炼为方法论: 步骤/决策点/分支 |
| 规范类 | 提炼为全局规范: 追加到 CLAUDE.md |

### 第四步: 质量检查

写入前必须通过:

- [ ] 内容来自对话中实际发生的事, 不是通用推断
- [ ] 包含具体的做法或代码, 不只是抽象概念
- [ ] 包含反模式或踩坑说明 (不能只写正确做法)
- [ ] 有明确的适用范围, 不是万能规则

不通过: 标注缺口, 补充后再检查

### 第五步: 展示确认

展示内容 + 蒸馏来源 + 建议存放位置, 用户确认后写入

## 载体与存放

所有路径相对于当前项目根目录

- **Rule** (`.claude/rules/<name>.md`): 纠正/踩坑/检查项, 简洁条目式, 可选 `paths` 限定范围
- **Skill** (`.claude/skills/<name>/SKILL.md`): 可复用工作流, frontmatter (name + description) + 方法论 (< 500 行)
- **CLAUDE.md** (追加, 不删除): 全局行为规范, `##` 分段, 与现有风格一致

## 知识组织模板

蒸馏产物应包含以下段落(按需选用):

- **学习记录**: 按日期, 什么有效 / 什么无效
- **已知问题**: 现象 / 原因 / 解决方法
- **反模式**: 禁止做法 / 为什么 / 正确做法
- **有效配置**: 经过测试的配置快照
