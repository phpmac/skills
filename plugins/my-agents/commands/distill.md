## 规范

- 只蒸馏方法/流程/规范/技巧/思路/经验
- 如果对话中没有可蒸馏内容, 告知用户暂无
- 禁止直接写入, 必须先展示完整内容经用户确认

## 蒸馏优先级

1. **纠正/踩坑** -> Rule 或 CLAUDE.md (最高优先, 防止再犯)
2. **self-evaluation 低分项** -> Rule 或 Skill (高优先, 补能力短板)
3. **可复用工作流** -> Skill
4. **全局行为规范** -> CLAUDE.md 追加

## 确定性分级

蒸馏前判断内容的确定性, 不同级别用不同策略:

| 级别 | 判断标准 | 策略 | 产出 |
|------|---------|------|------|
| 确定 | 实际执行中验证过 2 次以上 / 用户明确纠正过 | 精准提炼, 只写已验证有效的 | Rule / Skill / CLAUDE.md |
| 待验证 | 讨论过方法但未实际使用 / 只出现过 1 次 | 暂存内存, 标注待验证, 不写入规则文件 | Memory, 下次同类事件触发时升级 |
| 探索 | 新技术/新框架首次使用 / 不确定方案 | 保留多种可能性, 写试验记录 | Memory, 验证成熟后再蒸馏为 Rule |

**关键规则**: "待验证"和"探索"级别的内容禁止直接写入 Rule/CLAUDE.md, 必须先经过实际验证升级为"确定"后方可写入.

## 多源采集

从以下来源采集蒸馏输入, 按优先级排序:

| 来源 | 优先级 | 采集方式 |
|------|--------|----------|
| 用户纠正/踩坑 | 最高 | 回顾当前会话中用户纠正的内容和踩坑记录 |
| self-eval 低分项 | 高 | 检查会话中自评估维度 < 8 的产出 |
| 反复出现的模式 | 高 | 跨来源找交集: 同类问题出现 2 次以上 |
| memory 记录 | 中 | 读取 .claude/memory/ 中相关记忆文件 |
| git 最近变更 | 中 | git log --oneline -20 查看近期改动模式 |

**关键规则**: 同一问题在多个来源中出现, 蒸馏优先级自动提升.

## 领域分类

蒸馏前先判断领域, 不同领域用不同策略:

| 领域 | 判断依据 | 蒸馏重点 | 产出载体 |
|------|----------|----------|----------|
| code | 涉及代码编写/修改/重构 | 边界条件/性能/风格/DRY | Rule 或 Skill |
| security | 涉及漏洞/审计/修复 | 攻击面/验证方法/修复模式 | Rule |
| workflow | 涉及流程/协作/决策 | 流程优化/决策点/效率提升 | Skill 或 CLAUDE.md |
| infra | 涉及服务器/网络/容器 | 诊断路径/修复命令/预防措施 | Rule 或 Skill |

混合领域: 分别蒸馏, 产出到不同载体.

## 蒸馏流程

### 第一步: 多源采集

按上述 5 个来源逐一采集, 汇总为蒸馏候选列表:

```
采集结果:
  [来源] [领域] [内容摘要] [出现次数]
  ──────────────────────────────────
  [纠正] [security] 审计报告未区分已确认/疑似漏洞  1次
  [self-eval] [code] 准确性 6/10 - 缺少修改前后对比  1次
  [重复] [workflow] 方案未先画流程图就开始编码  2次 (优先级提升)
```

### 第二步: 深度提炼

对每个候选项进行根因追问:

- **发生了什么**: 具体事件描述
- **根因是什么**: 为什么会发生(不是表面原因)
- **正确做法是什么**: 基于根因的正确方法
- **反模式是什么**: 以后不能怎么做
- **如何预防**: 能否自动化检测/阻止

一次只问一个问题, 优先从踩坑/失败开始.

### 第三步: 领域匹配

根据领域分类, 选择蒸馏策略:

- **code**: 提炼为检查项/代码模式/性能规则, 含修改前后代码对比
- **security**: 提炼为攻击面清单/验证步骤/修复模板, 含 PoC 示例
- **workflow**: 提炼为流程步骤/决策树/检查清单, 含 ASCII 流程图
- **infra**: 提炼为诊断命令/修复脚本/预防配置, 含可执行命令

### 第四步: 质量门禁

写入前必须通过自动验证:

- [ ] 来源是实际任务执行中的行为/错误/纠正, 不是理论推断或通用最佳实践
- [ ] 如果来源是"讨论了一个方法但未实际使用", 标记为待验证, 不直接写入
- [ ] 包含具体的做法或代码, 不只是抽象概念
- [ ] 包含反模式或踩坑说明 (不能只写正确做法)
- [ ] 有明确的适用范围, 不是万能规则
- [ ] grep 检查不与现有规则重复 (如有重叠则合并而非新建)
- [ ] 包含修改前/修改后对比 (code/security 领域必须)

不通过: 标注缺口, 补充后再检查.

### 第五步: 自评估

对蒸馏产出写出评判过程:

- **已覆盖**: 本次蒸馏覆盖了采集列表中的哪些项?
- **可能遗漏**: 有哪些采集到的内容没有被蒸馏? 为什么?
- **不确定项**: 蒸馏内容中哪些是基于实际事件? 哪些是推断?
- **防再犯评估**: 这条规则/skill 真的能防止同类问题再发生吗? 有没有漏洞?

如果有明显遗漏或不确定项较多, 补充后重新蒸馏.

### 第六步: 展示确认

展示以下内容, 用户确认后写入:

```
蒸馏结果:
  来源: <哪些来源, 出现次数>
  领域: <code/security/workflow/infra>
  内容: <蒸馏内容>
  载体: <Rule/Skill/CLAUDE.md>
  路径: <建议存放路径>

自评估: <覆盖度>/<准确度>/<可操作性>/<防再犯> = 综合

确认写入?
```

## 载体与存放

所有路径相对于当前项目根目录

- **Rule** (`.claude/rules/<name>.md`): 纠正/踩坑/检查项, 简洁条目式, 可选 `paths` 限定范围
- **Skill** (`.claude/skills/<name>/SKILL.md`): 可复用工作流, frontmatter (name + description) + 方法论 (< 500 行)
- **CLAUDE.md** (追加, 不删除): 全局行为规范, `##` 分段, 与现有风格一致

## 知识组织模板

蒸馏产物应包含以下段落(按需选用):

- **学习记录**: 按日期, 什么有效 / 什么无效
- **已知问题**: 现象 / 原因 / 解决方法
- **反模式**: 禁止做法 / 为什么 / 正确做法
- **有效配置**: 经过测试的配置快照

## 自动触发建议

满足以下条件时, 主动建议用户执行蒸馏:

- self-evaluation 任意维度 < 8 且已补充完成
- 用户纠正了 3 次以上同类问题
- 一次会话中踩坑 > 2 次
- 会话涉及新技术/新框架的首次使用

建议话术: "本次会话有 N 个值得蒸馏的发现, 要执行 /distill 吗?"
