# PR Review Toolkit

全面的专业 Agent 集合, 用于彻底的 PR 审查, 涵盖代码注释, 测试覆盖, 错误处理, 类型设计, 代码质量和代码简化.

## 概述

此插件捆绑了 6 个专家审查 Agent, 每个 Agent 专注于代码质量的一个特定方面. 可单独用于针对性审查, 或一起用于全面的 PR 分析.

## Agents

### 1. comment-analyzer
**关注点**: 代码注释准确性和可维护性

**分析:**
- 注释与实际代码的准确性
- 文档完整性
- 注释腐烂和技术债务
- 误导性或过时的注释

**何时使用:**
- 添加文档后
- 最终确定带有注释变更的 PR 之前
- 审查现有注释时

**触发:**
```
"检查注释是否准确"
"审查我添加的文档"
"分析注释的技术债务"
```

### 2. pr-test-analyzer
**关注点**: 测试覆盖质量和完整性

**分析:**
- 行为覆盖 vs 行覆盖
- 测试覆盖中的关键缺口
- 测试质量和弹性
- 边界情况和错误条件

**何时使用:**
- 创建 PR 后
- 添加新功能时
- 验证测试彻底性

**触发:**
```
"检查测试是否彻底"
"审查此 PR 的测试覆盖"
"有没有关键的测试缺口?"
```

### 3. silent-failure-hunter
**关注点**: 错误处理和静默失败

**分析:**
- catch 块中的静默失败
- 不充分的错误处理
- 不适当的回退行为
- 缺少错误日志

**何时使用:**
- 实现错误处理后
- 审查 try/catch 块时
- 最终确定带有错误处理的 PR 之前

**触发:**
```
"审查错误处理"
"检查静默失败"
"分析此 PR 中的 catch 块"
```

### 4. type-design-analyzer
**关注点**: 类型设计质量和不变量

**分析:**
- 类型封装 (评分 1-10)
- 不变量表达 (评分 1-10)
- 类型有用性 (评分 1-10)
- 不变量强制 (评分 1-10)

**何时使用:**
- 引入新类型时
- 创建带有数据模型的 PR 时
- 重构类型设计时

**触发:**
```
"审查 UserAccount 类型设计"
"分析此 PR 中的类型设计"
"检查此类型是否有强不变量"
```

### 5. code-reviewer
**关注点**: 项目规则通用代码审查

**分析:**
- CLAUDE.md 合规性
- 风格违规
- Bug 检测
- 代码质量问题

**何时使用:**
- 编写或修改代码后
- 提交变更前
- 创建 PR 前

**触发:**
```
"审查我最近的变更"
"检查一切是否看起来正常"
"在我提交前审查此代码"
```

### 6. code-simplifier
**关注点**: 代码简化和重构

**分析:**
- 代码清晰度和可读性
- 不必要的复杂性和嵌套
- 冗余代码和抽象
- 与项目标准的一致性
- 过于紧凑或聪明的代码

**何时使用:**
- 编写或修改代码后
- 通过代码审查后
- 当代码工作但感觉很复杂时

**触发:**
```
"简化此代码"
"让这更清晰"
"改进此实现"
```

**注意**: 此 Agent 在改进代码结构和可维护性的同时保留功能.

## 使用模式

### 单独 Agent 使用

只需提出与 Agent 关注领域匹配的问题, Claude 会自动触发相应的 Agent:

```
"你能检查测试是否覆盖所有边界情况吗?"
→ 触发 pr-test-analyzer

"审查 API 客户端中的错误处理"
→ 触发 silent-failure-hunter

"我添加了文档 - 准确吗?"
→ 触发 comment-analyzer
```

### 全面 PR 审查

对于彻底的 PR 审查, 请求多个方面:

```
"我准备创建此 PR. 请:
1. 审查测试覆盖
2. 检查静默失败
3. 验证代码注释准确
4. 审查任何新类型
5. 通用代码审查"
```

这将触发所有相关 Agent 分析 PR 的不同方面.

### 主动审查

Claude 可能根据上下文主动使用这些 Agent:

- **编写代码后** → code-reviewer
- **添加文档后** → comment-analyzer
- **创建 PR 前** → 多个 Agent 视情况而定
- **添加类型后** → type-design-analyzer

## Agent 详情

### 置信度评分

Agent 为其发现提供置信度评分:

**comment-analyzer**: 在准确性检查中以高置信度识别问题

**pr-test-analyzer**: 评分测试缺口 1-10 (10 = 关键, 必须添加)

**silent-failure-hunter**: 标记错误处理问题的严重性

**type-design-analyzer**: 在 4 个维度上评分 1-10

**code-reviewer**: 评分问题 0-100 (91-100 = 关键)

**code-simplifier**: 识别复杂性并建议简化

### 输出格式

所有 Agent 提供结构化, 可操作的输出:
- 清晰的问题识别
- 具体的文件和行引用
- 解释为什么是问题
- 改进建议
- 按严重性优先排序

## 最佳实践

### 何时使用每个 Agent

**提交前:**
- code-reviewer (通用质量)
- silent-failure-hunter (如果修改了错误处理)

**创建 PR 前:**
- pr-test-analyzer (测试覆盖检查)
- comment-analyzer (如果添加/修改了注释)
- type-design-analyzer (如果添加/修改了类型)
- code-reviewer (最终扫视)

**通过审查后:**
- code-simplifier (改进清晰度和可维护性)

**PR 审查期间:**
- 任何 Agent 用于提出的具体关注
- 修复后针对性重新审查

### 运行多个 Agent

您可以请求多个 Agent 并行或顺序运行:

**并行** (更快):
```
"并行运行 pr-test-analyzer 和 comment-analyzer"
```

**顺序** (当一个为另一个提供信息时):
```
"首先审查测试覆盖, 然后检查代码质量"
```

## 技巧

- **要具体**: 针对特定 Agent 进行聚焦审查
- **主动使用**: 在创建 PR 前运行, 不是之后
- **先解决关键问题**: Agent 优先排序发现
- **迭代**: 修复后再次运行以验证
- **不要过度使用**: 关注变更的代码, 不是整个代码库

## 故障排除

### Agent 未触发

**问题**: 请求审查但 Agent 没有运行

**解决方案**:
- 在请求中更具体
- 明确提及 Agent 类型
- 引用具体关注 (例如, "测试覆盖")

### Agent 分析错误文件

**问题**: Agent 审查太多或错误的文件

**解决方案**:
- 指定要关注的文件
- 引用 PR 号或分支
- 提及 "最近的变更" 或 "git diff"

## 与工作流集成

此插件与以下配合良好:
- **build-validator**: 在审查前运行构建/测试
- **项目特定 Agent**: 与您的自定义 Agent 结合

**推荐工作流:**
1. 编写代码 → **code-reviewer**
2. 修复问题 → **silent-failure-hunter** (如果错误处理)
3. 添加测试 → **pr-test-analyzer**
4. 文档 → **comment-analyzer**
5. 审查通过 → **code-simplifier** (优化)
6. 创建 PR
