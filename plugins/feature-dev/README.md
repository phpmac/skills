# Feature Development 插件

一个全面, 结构化的功能开发工作流, 配备专业 Agent 用于代码库探索, 架构设计和质量审查.

## 概述

Feature Development 插件提供系统的 7 阶段方法来构建新功能. 它不是直接跳入代码, 而是引导您理解代码库, 提出澄清问题, 设计架构并确保质量 - 从而产生更好设计的功能, 与现有代码无缝集成.

## 理念

构建功能不仅仅是编写代码. 您需要:
- **理解代码库** 在做修改之前
- **提问问题** 澄清模糊的需求
- **深思熟虑地设计** 在实现之前
- **审查质量** 在构建之后

此插件将这些实践嵌入到结构化工作流中, 当您使用 `/feature-dev` 命令时自动运行.

## 命令: `/feature-dev`

启动包含 7 个不同阶段的引导式功能开发工作流.

**用法:**
```bash
/feature-dev Add user authentication with OAuth
```

或简单地:
```bash
/feature-dev
```

该命令将以交互方式引导您完成整个过程.

## 7 阶段工作流

### 阶段 1: 探索 (Discovery)

**目标**: 理解需要构建什么

**发生什么:**
- 如果功能请求不清楚则进行澄清
- 询问您要解决什么问题
- 识别约束和要求
- 总结理解并与您确认

**示例:**
```
You: /feature-dev Add caching
Claude: 让我理解您的需求...
        - 应该缓存什么? (API 响应, 计算值等)
        - 您的性能要求是什么?
        - 您有首选的缓存方案吗?
```

### 阶段 2: 代码库探索

**目标**: 理解相关的现有代码和模式

**发生什么:**
- 并行启动 2-3 个 `code-explorer` agent
- 每个 agent 探索不同方面 (类似功能, 架构, UI 模式)
- Agent 返回带有要阅读的关键文件的全面分析
- Claude 阅读所有识别的文件以建立深入理解
- 展示发现的综合摘要

**启动的 Agent:**
- "查找与 [功能] 类似的功能并追踪实现"
- "映射 [区域] 的架构和抽象"
- "分析 [相关功能] 的当前实现"

**示例输出:**
```
发现类似功能:
- 用户认证 (src/auth/): 使用 JWT token, 中间件模式
- 会话管理 (src/session/): Redis 支持, 24小时过期
- API 安全 (src/api/middleware/): 速率限制, CORS

理解的关键文件:
- src/auth/AuthService.ts:45 - 核心认证逻辑
- src/middleware/authMiddleware.ts:12 - 请求认证
- src/config/security.ts:8 - 安全配置
```

### 阶段 3: 澄清问题

**目标**: 填补空白并解决所有歧义

**发生什么:**
- 审查代码库发现和功能请求
- 识别未明确的方面:
  - 边界情况
  - 错误处理
  - 集成点
  - 向后兼容性
  - 性能需求
- 以有组织的列表呈现所有问题
- **等待您的回答后再继续**

**示例:**
```
在设计架构之前, 我需要澄清:

1. OAuth 提供商: 哪些 OAuth 提供商? (Google, GitHub, 自定义?)
2. 用户数据: 存储 OAuth token 还是仅用户资料?
3. 现有认证: 替换当前认证还是并行添加?
4. 会话: 与现有会话管理集成?
5. 错误处理: 如何处理 OAuth 失败?
```

**关键**: 此阶段确保设计开始前没有任何歧义.

### 阶段 4: 架构设计

**目标**: 设计多种实现方案

**发生什么:**
- 启动 2-3 个 `code-architect` agent, 关注点不同:
  - **最小变更**: 最小的改动, 最大复用
  - **清洁架构**: 可维护性, 优雅抽象
  - **务实平衡**: 速度 + 质量
- 审查所有方案
- 形成对哪个最适合此任务的意见
- 展示比较, 权衡和建议
- **询问您更喜欢哪种方案**

**示例输出:**
```
我设计了 3 种方案:

方案 1: 最小变更
- 用 OAuth 方法扩展现有 AuthService
- 向现有 auth router 添加新 OAuth 路由
- 需要最少的重构
优点: 快速, 低风险
缺点: OAuth 与现有认证耦合, 更难测试

方案 2: 清洁架构
- 新 OAuthService 具有专用接口
- 独立的 OAuth router 和中间件
- 重构 AuthService 使用公共接口
优点: 清晰分离, 可测试, 可维护
缺点: 更多文件, 更多重构

方案 3: 务实平衡
- 新 OAuthProvider 抽象
- 集成到现有 AuthService
- 最少重构, 良好边界
优点: 平衡复杂性和清洁度
缺点: 仍有一些耦合

推荐: 方案 3 - 为您提供清晰的边界而无需
过度重构, 并很好地适应您现有架构.

您想使用哪种方案?
```

### 阶段 5: 实现

**目标**: 构建功能

**发生什么:**
- **等待明确批准** 后再开始
- 阅读前几个阶段识别的所有相关文件
- 按选定的架构实现
- 严格遵循代码库约定
- 编写干净, 文档良好的代码
- 随着进展更新 todo

**注意:**
- 实现只在您批准后开始
- 遵循阶段 2 发现的模式
- 使用阶段 4 设计的架构
- 持续跟踪进度

### 阶段 6: 质量审查

**目标**: 确保代码简单, DRY, 优雅且功能正确

**发生什么:**
- 并行启动 3 个 `code-reviewer` agent, 关注点不同:
  - **简洁性/DRY/优雅**: 代码质量和可维护性
  - **Bug/正确性**: 功能正确性和逻辑错误
  - **约定/抽象**: 项目标准和模式
- 整合发现
- 识别最高严重性问题
- **展示发现并询问您想做什么**:
  - 现在修复
  - 稍后修复
  - 按原样继续
- 根据您的决定处理问题

**示例输出:**
```
代码审查结果:

高优先级问题:
1. OAuth 回调缺少错误处理 (src/auth/oauth.ts:67)
2. 内存泄漏: OAuth state 未清理 (src/auth/oauth.ts:89)

中优先级:
1. 可以简化 token 刷新逻辑 (src/auth/oauth.ts:120)
2. 考虑提取 OAuth 配置验证

所有测试通过. 代码遵循项目约定.

您想怎么做?
```

### 阶段 7: 总结

**目标**: 记录完成的工作

**发生什么:**
- 标记所有 todo 完成
- 总结:
  - 构建了什么
  - 做出的关键决策
  - 修改的文件
  - 建议的后续步骤

**示例:**
```
功能完成: OAuth 认证

构建了什么:
- 支持 Google 和 GitHub 的 OAuth 提供商抽象
- 与现有认证集成的 OAuth 路由和中间件
- Token 刷新和会话集成
- 所有 OAuth 流程的错误处理

关键决策:
- 使用带有 OAuthProvider 抽象的务实方案
- 与现有会话管理集成
- 添加 OAuth state 防止 CSRF

修改的文件:
- src/auth/OAuthProvider.ts (新)
- src/auth/AuthService.ts
- src/routes/auth.ts
- src/middleware/authMiddleware.ts

建议的后续步骤:
- 为 OAuth 流程添加测试
- 添加更多 OAuth 提供商 (Microsoft, Apple)
- 更新文档
```

## Agents

### `code-explorer`

**目的**: 通过追踪执行路径深入分析现有代码库功能

**关注领域:**
- 入口点和调用链
- 数据流和转换
- 架构层和模式
- 依赖和集成
- 实现细节

**何时触发:**
- 阶段 2 中自动触发
- 探索代码时可手动调用

**输出:**
- 带有 file:line 引用的入口点
- 逐步执行流程
- 关键组件和职责
- 架构见解
- 要阅读的基本文件列表

### `code-architect`

**目的**: 设计功能架构和实现蓝图

**关注领域:**
- 代码库模式分析
- 架构决策
- 组件设计
- 实现路线图
- 数据流和构建序列

**何时触发:**
- 阶段 4 中自动触发
- 可手动调用进行架构设计

**输出:**
- 发现的模式和约定
- 带有理由的架构决策
- 完整的组件设计
- 带有特定文件的实现映射
- 带有阶段的构建序列

### `code-reviewer`

**目的**: 审查代码的 bug, 质量问题和项目约定

**关注领域:**
- 项目规则合规性 (CLAUDE.md)
- Bug 检测
- 代码质量问题
- 基于置信度的过滤 (只报告高置信度问题 >=80)

**何时触发:**
- 阶段 6 中自动触发
- 编写代码后可手动调用

**输出:**
- 关键问题 (置信度 75-100)
- 重要问题 (置信度 50-74)
- 带有 file:line 引用的具体修复
- 项目规则引用

## 使用模式

### 完整工作流 (新功能推荐):
```bash
/feature-dev Add rate limiting to API endpoints
```

让工作流引导您完成所有 7 个阶段.

### 手动 Agent 调用:

**探索功能:**
```
"启动 code-explorer 追踪认证如何工作"
```

**设计架构:**
```
"启动 code-architect 设计缓存层"
```

**审查代码:**
```
"启动 code-reviewer 检查我最近的变更"
```

## 最佳实践

1. **对复杂功能使用完整工作流**: 7 个阶段确保彻底规划
2. **认真回答澄清问题**: 阶段 3 防止未来混淆
3. **慎重选择架构**: 阶段 4 给您选项是有原因的
4. **不要跳过代码审查**: 阶段 6 在问题进入生产前捕获它们
5. **阅读建议的文件**: 阶段 2 识别关键文件 - 阅读它们以理解上下文

## 何时使用此插件

**适用于:**
- 涉及多个文件的新功能
- 需要架构决策的功能
- 与现有代码的复杂集成
- 需求不太明确的功能

**不适用于:**
- 单行 bug 修复
- 琐碎变更
- 定义明确, 简单的任务
- 紧急热修复

## 要求

- 已安装 Claude Code
- Git 仓库 (用于代码审查)
- 具有现有代码库的项目 (工作流假设有现有代码可供学习)

## 故障排除

### Agent 耗时过长

**问题**: 代码探索或架构 Agent 缓慢

**解决方案**:
- 对于大型代码库这是正常的
- Agent 尽可能并行运行
- 彻底性会带来更好的理解

### 澄清问题过多

**问题**: 阶段 3 问太多问题

**解决方案**:
- 在初始功能请求中更具体
- 提前提供关于约束的上下文
- 如果真的没有偏好说 "whatever you think is best"

### 架构选项令人困惑

**问题**: 阶段 4 架构选项太多

**解决方案**:
- 信任推荐 - 它基于代码库分析
- 如果仍不确定, 要求更多解释
- 有疑问时选择务实选项

## 技巧

- **在功能请求中具体**: 更多细节 = 更少澄清问题
- **信任流程**: 每个阶段建立在前一个阶段之上
- **审查 Agent 输出**: Agent 提供关于您代码库的有价值见解
- **不要跳过阶段**: 每个阶段都有目的
- **用于学习**: 探索阶段教您关于自己的代码库
