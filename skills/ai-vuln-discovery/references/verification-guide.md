# 漏洞验证方法论

## 验证的核心原则 (强制执行)

**"断言漏洞存在的唯一方式是成功利用"**

**未经验证的猜测不是漏洞发现**: 代码分析只能发现"可能的漏洞",必须通过实际验证确认.没有PoC的假设只是猜测,不是安全发现.

### 验证清单 (必须全部完成)

| 检查项 | 要求 | 证明方式 |
|--------|------|----------|
| **漏洞触发** | 能稳定触发异常行为 | 复现步骤,截图/日志 |
| **影响证明** | 证明实际安全风险 | 数据访问,权限提升等 |
| **环境独立** | 不依赖特定开发环境 | 干净的Docker环境 |
| **可重复** | 多次执行结果一致 | 至少3次成功复现 |

### 禁止行为

以下情况**不能**标记为已验证漏洞:
- 仅通过代码分析推测存在,未实际触发
- 需要极端条件才能触发(如管理员手动修改数据库)
- 依赖未公开的内网服务
- 利用链存在断点,无法完整利用

## 验证环境选择

根据技术栈选择最适合的验证环境:

| 技术栈 | 推荐环境/工具 | 测试能力 |
|--------|---------------|----------|
| **Web应用(PHP/Python/Node等)** | Docker容器 | 隔离运行环境,模拟生产配置 |
| **Solidity智能合约** | Foundry/Forge | 单元测试,模糊测试,状态模拟 |
| **Java应用** | Maven/Gradle + TestContainers | 依赖管理,集成测试 |
| **Rust程序** | Cargo + 自定义测试框架 | 内存安全测试,并发测试 |
| **移动应用** | 模拟器/真机 + Frida | 运行时分析,动态调试 |

**环境选择原则**:
- 选择能最准确模拟目标运行环境的工具
- 优先使用目标生态的标准测试框架
- 确保环境可复现,可版本控制

## 验证流程

| 步骤 | 目标 | 关键活动 |
|------|------|----------|
| **环境搭建** | 创建可复现的测试环境 | 根据技术栈选择工具,配置依赖 |
| **漏洞触发** | 构造触发漏洞的输入 | 编写测试用例,准备Payload |
| **结果验证** | 确认漏洞被成功利用 | 断言验证,状态检查,日志分析 |
| **修复验证** | 确认修复后漏洞不可用 | 回归测试,边界条件测试 |

## 漏洞分析流程

```
开始审计
    |
    v
+-------------------------------+
| 1. 框架学习与环境准备          |
|    - 阅读官方文档理解标准用法  |
|    - 搭建适合技术栈的验证环境  |
+---------------+---------------+
                |
                v
+-------------------------------+
| 2. 版本与历史漏洞分析          |
|    - 确认框架精确版本          |
|    - 查询该版本的CVE和修复     |
+---------------+---------------+
                |
                v
+-------------------------------+
| 3. 代码差异分析                |
|    - 对比标准写法与实际写法    |
|    - 识别业务逻辑漏洞点        |
+---------------+---------------+
                |
                v
+-------------------------------+
| 4. 漏洞验证                    |
|    - 选择合适环境/工具         |
|    - 编写PoC脚本/测试用例      |
|    - 执行验证并记录结果        |
+---------------+---------------+
                |
                v
+-------------------------------+
| 5. 报告与修复验证              |
|    - 输出漏洞详情和修复建议    |
|    - 验证修复有效性            |
+-------------------------------+
```
