# 智能合约安全审计方法论

## 审计原则

**方法论优先**: 本文档提供审计方法论和学习路径,具体工具使用请查阅官方文档.

**适配项目框架**: 漏洞验证必须使用项目现有的测试框架,而非强制使用特定工具.

**学习资源**: 审计前应先学习已知漏洞模式,使用爬虫获取以下漏洞仓库进行分析.

审计前必须通过以下资源学习已知漏洞模式:

4. 对比漏洞代码与安全代码的差异

## 静态分析要求

### Slither (必须执行)

**要求**: 任何智能合约审计必须先执行 Slither 静态分析.

**分析重点**:
- 重入漏洞检测
- 访问控制缺陷
- 未初始化状态变量
- 外部调用安全问题
- 数据流和控制流分析

**输出解读**:
- Impact: 漏洞影响程度
- Confidence: 检测置信度
- 优先处理 Impact=High 且 Confidence=High 的发现

**具体使用方法**: 使用 Context7 或网络爬虫查询 Slither 官方文档

### Mythril (建议执行)

**用途**: 符号执行分析,适合深度分析复杂逻辑.

**具体使用方法**: 使用 Context7 或网络爬虫查询 Mythril 官方文档

## 漏洞验证: 适配项目框架

### 核心原则

**单元测试验证漏洞的优势**:
- 离线测试,不需要网络
- 自部署测试,可以自己部署合约
- 可复现,快速迭代

**强制要求**: 漏洞验证必须编写单元测试,只有通过测试验证的漏洞才是真漏洞.

### 框架识别方法

1. 查看项目目录结构和配置文件
2. 网络搜索识别框架名称和版本
3. 使用 Context7 查询该框架测试文档

### 编写测试

- 使用项目现有框架编写测试
- 参考项目现有测试结构
- 注意框架版本差异

### 测试编写要求

无论使用哪种框架,漏洞验证测试必须包含:

1. **漏洞触发**: 构造能触发漏洞的输入
2. **状态断言**: 验证漏洞导致的状态变化
3. **影响证明**: 证明漏洞的实际安全影响
4. **可复现性**: 测试可重复执行

## 漏洞分析要点

### 高危漏洞类型

| 漏洞类型 | 分析要点 |
|----------|----------|
| **重入攻击** | 外部调用前后的状态变更顺序 |
| **访问控制** | 敏感函数的权限检查 |
| **价格操控** | 价格来源和操纵可能性 |
| **闪电贷攻击** | 依赖外部状态的业务逻辑 |
| **delegatecall** | 代理合约的可控性 |

### 分析方法论

1. **理解业务逻辑**: 先理解合约要实现什么功能
2. **识别信任边界**: 哪些输入是不可信的
3. **追踪资金流向**: ETH/Token 的转入转出
4. **分析权限模型**: 谁能做什么操作
5. **检查外部依赖**: 预言机,外部合约调用

## DeFi 特有漏洞

| 攻击类型 | 分析方法 |
|----------|----------|
| **价格操纵** | 检查价格来源是否可被操纵 |
| **闪电贷攻击** | 分析单区块内多操作的可行性 |
| **三明治攻击** | 检查交易排序依赖 |
| **预言机攻击** | 验证预言机数据的可信度 |

**学习方法**: 分析 DeFiHackLabs 中的真实攻击案例.

## 关键原则

1. **先学习后审计**: 通过漏洞仓库学习已知模式
2. **静态分析先行**: 必须先用 Slither 扫描
3. **适配项目框架**: 使用项目现有框架编写测试
4. **单元测试验证**: 只有通过测试验证的漏洞才是真漏洞
5. **版本敏感**: 同一代码在不同 Solidity 版本可能有不同风险
